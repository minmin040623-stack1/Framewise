# FrameWise 발표 자료

> 사진 구도를 직접 연습하고, 필름 색감을 비교하는 브라우저 기반 사진 학습 도구

---

## 0. 발표 핵심 요약

FrameWise는 처음에는 **사진을 직접 크롭하고 구도 점수를 받는 게임**으로 시작했다.
이후 구도 이론 조사, 점수 체계 검토, 사용자 피드백을 반복하며 설명 가능한
학습 도구로 발전했다. 마지막에는 기존 기능을 보존한 채 **AI Film Lab**을
별도 기능으로 추가했다.

이 과정에서:

- 사람은 프로젝트 방향, 사진의 적절성, 구도 판단, 문장과 UI의 최종 선택을 맡았다.
- ChatGPT는 아이디어 확장과 요구사항·프롬프트 정리에 활용했다.
- Codex는 VS Code의 실제 프로젝트를 읽고 조사, 코드 수정, 테스트, Git 작업을 수행했다.
- 웹사이트 내부의 점수와 필름 추천은 생성형 AI의 감상평이 아니라
  **사람이 정한 기준과 규칙 기반 계산**으로 만들어졌다.

발표용 한 문장:

> FrameWise는 AI에게 정답을 맡긴 프로젝트가 아니라, 사람이 사진의 기준을 설계하고
> AI와 함께 구현·검증의 반복 속도를 높인 프로젝트입니다.

# 1부. 발표 슬라이드 구성

## 슬라이드 1. FrameWise 소개

### 화면에 넣을 내용

- 사진 구도를 직접 크롭하며 배우는 `Composition Challenge`
- 업로드한 사진의 필름 색감을 비교하는 `AI Film Lab`
- 별도의 서버나 설치 없이 브라우저에서 작동

### 발표 멘트

FrameWise는 사진을 보는 감각을 직접 연습하기 위해 만든 웹사이트입니다.
사용자가 사진을 자르고 끝나는 것이 아니라, 왜 이런 구도가 더 설득력 있는지
점수와 선, 문장으로 확인할 수 있습니다. 이후 같은 사진에 여러 필름 분위기를
적용해 비교하는 Film Lab도 추가했습니다.

## 슬라이드 2. 프로젝트를 시작한 이유

### 문제 인식

- 완성된 사진만 보면 크롭 전후의 차이를 체감하기 어렵다.
- “좋은 구도”라는 말은 주관적으로 들리기 쉽다.
- 점수만 보여 주면 사용자가 결과를 납득하기 어렵다.
- 하나의 정답 크롭만 제시하면 사진가의 다양한 의도를 무시할 수 있다.

### 목표

- 사용자가 직접 움직여 보며 구도 차이를 체감하게 한다.
- 사진마다 알맞은 구도 기준만 선택해 평가한다.
- 점수의 이유를 좌표, 비율, 가이드 선과 문장으로 설명한다.
- 여러 추천 크롭을 보여 주어 정답이 하나가 아님을 알려 준다.

## 슬라이드 3. 전체 제작 흐름

### 1단계 — 프로젝트 기준 정리

1. 기존 HTML, CSS, JavaScript와 `PROJECT.md`를 확인했다.
2. 현재 가능한 기능과 불가능한 기능을 구분했다.
3. 구도 점수, 데이터 구조, 저작권, UI 원칙을 문서로 정리했다.

### 2단계 — Composition Challenge 구현

1. 사진 선택과 크롭 기능을 구현했다.
2. 사진별 목표 구도와 수동 좌표 주석을 `photos.json`에 저장했다.
3. 점수 계산, 결과 근거, 복수 추천 크롭, 가이드 선을 추가했다.
4. 연속으로 같은 사진이 나오지 않는 선택 규칙을 만들었다.

### 3단계 — 검토와 개선

1. 실제 사용자 크롭과 추천 크롭을 비교했다.
2. 비슷한 크롭이 지나치게 낮은 점수를 받는 문제를 발견했다.
3. 중복 감점과 사진별 허용 범위를 조정했다.
4. 한국어 문장, 줄바꿈, 이미지 출처와 라이선스 표시를 개선했다.

### 4단계 — 새로운 기능 제안

1. 기존 안정 버전을 태그와 브랜치로 보존했다.
2. 별도 페이지인 `AI Film Lab`을 제안했다.
3. 영상 비교 자료와 공식 제품 설명을 조사했다.
4. 8개의 독자적인 `Inspired` 프리셋을 구현했다.
5. 업로드, 추천, 비교 슬라이더, 강도 조절, 다운로드를 완성했다.

### 흐름 요약

```text
PROJECT.md와 기존 코드 확인
        ↓
기능 요구사항과 제한 정의
        ↓
작은 기능 구현
        ↓
사용자 화면 확인과 피드백
        ↓
점수·문장·UI 수정
        ↓
테스트와 Git 저장
        ↓
새 기능 제안 후 별도 브랜치에서 확장
```

## 슬라이드 4. `PROJECT.md`를 중심으로 작업한 이유

`PROJECT.md`는 단순한 소개문이 아니라 프로젝트의 **공통 기준서**로 사용했다.

- 두 기능의 사용자 흐름
- 현재 구현된 기능과 아직 없는 기능
- 구도 점수의 원칙과 데이터 구조
- Film Lab의 처리 방식과 성능 제한
- “AI가 자동 인식했다”고 과장하지 않기 위한 표현 원칙
- 저작권과 출처 정책
- 테스트와 다음 우선순위

### 실제 작업 방식

```text
PROJECT.md 확인
    → 현재 구조에 맞는 변경 계획
    → HTML/CSS/JS 구현
    → 테스트
    → 구현 결과를 PROJECT.md에 다시 반영
```

새로운 대화나 작업이 시작되어도 `PROJECT.md`를 먼저 읽으면 프로젝트의 목적과
금지 사항을 잃지 않고 이어서 작업할 수 있었다.

## 슬라이드 5. 사람과 AI의 역할 분담

| 구분 | 맡은 역할 |
|---|---|
| 프로젝트 제작자 | 문제 정의, 사용할 사진 선택, 직접 촬영 사진의 권리 확인, 구도 판단, UI 피드백, 최종 결정 |
| ChatGPT | 아이디어 확장, 구도 이론 목록화, 요구사항을 긴 작업 프롬프트로 정리, 새 기능 방향 탐색 |
| Codex | 프로젝트 파일 분석, 웹 자료 조사, 코드 구현·수정, 오류 진단, 테스트 작성·실행, Git 브랜치·커밋·푸시 |
| 웹사이트의 규칙 엔진 | 수동 주석과 수학적 기준으로 구도 점수 계산 |
| Film Lab 분석기 | 밝기·색·대비·질감 통계를 이용해 규칙 기반으로 프리셋 추천 |

### 가장 중요한 구분

- ChatGPT와 Codex는 **제작 과정의 AI 도구**다.
- Composition Challenge가 사용자의 사진을 AI 모델로 평가하는 것은 아니다.
- Film Lab도 현재 생성형 AI나 머신러닝 모델을 사용하지 않는다.
- 자동 피사체 인식 대신 사람이 검토한 좌표를 사용한다.
- Film Lab의 “추천”은 평균 밝기, 채도, 색온도 등의 수치에 따른 규칙이다.

## 슬라이드 6. ChatGPT를 활용한 과정

### 1. 아이디어 발산

- 어떤 사진 구도를 학습 대상으로 삼을지 목록화했다.
- 삼등분할, 중앙, 대칭, 리딩 라인, 시선 여백, 전·중·배경 등으로 범위를 넓혔다.
- 크롭 전후가 크게 달라지는 사진의 조건을 고민했다.

### 2. 요구사항 구체화

- “사진마다 다른 목표 구도를 평가한다.”
- “하나의 정답 크롭만 강요하지 않는다.”
- “가능한 기능과 아직 불가능한 기능을 구분한다.”
- “출처와 라이선스를 함께 관리한다.”

이 생각들을 긴 작업 프롬프트로 정리하여 Codex가 실제 프로젝트에 적용할 수 있게 했다.

### 3. 새 기능 제안

- 기존 크롭 게임과 별개의 Film Lab 흐름을 구상했다.
- 업로드 → 프리셋 비교 → 강도 조절 → 원본 비교 → 다운로드 순서를 정했다.
- 실제 필름을 정확히 복제한다는 표현 대신 모든 이름에 `Inspired`를 붙이도록 했다.

## 슬라이드 7. Codex를 활용한 과정

### 1. 프로젝트 이해

- `PROJECT.md`, `TASK.md`, 기존 HTML/CSS/JS, `photos.json`을 읽었다.
- 기존 기능과 충돌할 부분을 먼저 확인했다.

### 2. 조사

- Canon, Adobe 등 공식 자료를 바탕으로 구도 기준을 정리했다.
- 재사용 가능한 사진 후보와 라이선스를 조사했다.
- Film Lab 참고 영상의 순서와 타임스탬프를 확인했다.
- 제작자가 제공한 비교 화면을 보고 상대적인 색 차이를 정리했다.

### 3. 구현

- 사진별 점수 엔진과 복수 추천 크롭을 구현했다.
- 결과 화면의 가이드 선과 설명을 만들었다.
- 한국어 UI와 Image Credits 페이지를 추가했다.
- WebGL과 Canvas2D를 이용한 Film Lab을 구현했다.

### 4. 검증과 수정

- 추천 크롭이 실제로 높은 점수를 받는지 테스트했다.
- 비슷한 크롭에 중복 감점이 발생하는 문제를 수정했다.
- 비교 슬라이더의 라벨, 경계선, 손잡이 방향을 일치시켰다.
- 전체 10개 테스트 파일을 반복 실행했다.

### 5. 버전 관리

- 안정 버전을 태그로 보존했다.
- Film Lab은 `feature/film-lab` 브랜치에서 개발했다.
- 변경 단위마다 커밋하고 GitHub에 푸시했다.

## 슬라이드 8. VS Code와 AI 이외의 도구 활용

### VS Code

- 파일 탐색기로 HTML, CSS, JS, JSON, Markdown 구조를 관리했다.
- 코드 편집 화면에서 수정된 위치를 직접 확인했다.
- 브라우저 결과와 코드를 나란히 비교하며 UI를 조정했다.
- 터미널을 열어 테스트와 Git 명령을 실행했다.

### VS Code 터미널

- 로컬 서버 실행
- 테스트 실행
- 현재 브랜치와 변경 파일 확인
- 커밋과 GitHub 푸시

### Git과 GitHub

- 기능별 변경 이력을 남겼다.
- 안정 버전과 실험 기능을 브랜치로 분리했다.
- 문제가 생겨도 이전 버전으로 돌아갈 수 있게 했다.
- 팀원이 같은 버전을 확인할 수 있도록 원격 저장소에 공유했다.

### 브라우저 개발 흐름

- 실제 화면에서 크롭, 점수, 비교 슬라이더, 반응형 문장을 확인했다.
- 사용자가 발견한 어색한 점을 스크린샷으로 다시 검토했다.
- 코드 수정 → 새로고침 → 사용자 피드백의 짧은 반복을 사용했다.

## 슬라이드 9. HTML, CSS, JavaScript의 역할

| 파일 종류 | 역할 | FrameWise 예시 |
|---|---|---|
| HTML | 화면의 구조와 의미를 만든다. | 제목, 버튼, 카드, 업로드 영역, 캔버스, 접근성 속성 |
| CSS | 화면의 모양과 배치를 정한다. | 색상, 간격, 반응형 화면, 카드, 슬라이더, 가이드 UI |
| JavaScript | 사용자의 행동에 반응하고 데이터를 계산한다. | 사진 선택, 타이머, 크롭, 점수 계산, 필름 렌더링, 다운로드 |
| JSON | 코드와 화면이 사용할 콘텐츠 데이터를 저장한다. | 사진별 미션, 좌표, 점수 가중치, 추천 크롭, 출처 |
| Markdown | 프로젝트의 목표와 조사 근거를 기록한다. | `PROJECT.md`, 구도 가이드, 필름 참고 문서 |

### 쉬운 비유

- HTML은 건물의 **뼈대와 방 배치**
- CSS는 건물의 **인테리어와 동선**
- JavaScript는 버튼과 장치가 작동하는 **전기·제어 시스템**
- JSON은 화면에 넣을 내용을 보관하는 **자료 창고**
- Markdown은 팀이 함께 보는 **설계서와 작업 기록**

## 슬라이드 11. Composition Challenge의 작동 방식

```text
photos.json
  사진·미션·수동 주석·가중치·추천 크롭
        ↓
game.js + Cropper.js
  사진 선택·타이머·사용자 크롭
        ↓
composition-score.js
  사진에 지정된 기준만 계산
        ↓
LocalStorage
  크롭 이미지·점수·사진 정보 전달
        ↓
result.js
  점수 근거·가이드 선·복수 추천 예시 표시
```

### 점수 설계 원칙

- 모든 사진에 같은 규칙을 적용하지 않는다.
- 목표로 지정된 구도를 따르지 않았을 때만 관련 점수에 반영한다.
- 목표 구도, 피사체 보존, 정렬, 여백과 균형, 추천 크롭 유사도를 분리한다.
- 추천 크롭 유사도는 보조 신호이며 전체 정답이 아니다.
- 피사체와 선의 위치는 자동 인식이 아니라 수동 주석이다.

### 현재 수치

- 연습 사진 13장
- 직접 촬영 6장
- Creative Commons 자료 7장
- 모든 사진에 복수 추천 크롭과 구체적인 이유 제공

---

## 슬라이드 12. 점수를 납득할 수 있게 만든 방법

### 처음 발견한 문제

- 사용자의 크롭이 추천안과 거의 비슷해도 52점처럼 낮게 나오는 경우가 있었다.
- 비슷한 원인으로 여러 항목에서 반복 감점되는 문제가 있었다.
- 하나의 추천 사각형과 조금만 달라도 과도하게 점수가 낮아질 수 있었다.

### 개선

- 사진별 목표에 따라 허용 범위를 다르게 설정했다.
- 추천 크롭과 충분히 비슷하면 중복 감점을 완화했다.
- 모든 추천 크롭이 같은 점수 엔진에서 85점 이상인지 테스트했다.
- 추천 크롭이 원본 그대로보다 더 높은 점수를 받는지 확인했다.
- 세부 점수와 측정 근거를 함께 보여 주었다.

### 발표에서 강조할 점

> 점수의 목적은 미적 우열을 단정하는 것이 아니라, 이번 사진에서 연습하려는
> 구도 목표와 사용자의 선택이 얼마나 가까운지 설명하는 것입니다.

## 슬라이드 13. Film Lab을 새 기능으로 추가한 과정

### 왜 별도 기능으로 만들었나

- 크롭은 프레임을 바꾸는 연습이다.
- 필름 색감은 같은 프레임에서 분위기가 어떻게 달라지는지 비교하는 연습이다.
- 목적이 다르기 때문에 기존 게임을 바꾸지 않고 독립 페이지로 추가했다.

### 사용자 흐름

```text
사진 업로드
    ↓
작은 미리보기로 밝기·색·대비 분석
    ↓
규칙 기반 프리셋 한 개 추천
    ↓
8개 Inspired 스타일 동시 비교
    ↓
효과·입자 강도 조절
    ↓
원본/결과 슬라이더 비교
    ↓
JPG 또는 PNG 다운로드
```

### 구현한 프리셋

- Portra 400 Inspired
- Fuji C200 Inspired
- CineStill 400D Inspired
- Lomography CN100 Inspired
- Harman Phoenix 200 Inspired
- LomoChrome Purple Inspired
- RETO Aqua 400 Inspired
- B&W 400 Inspired

### 정직한 표현

- 공식 LUT나 유료 프로필을 사용하지 않았다.
- 영상을 그대로 복사하지 않고 상대적 색 차이만 참고했다.
- 모든 프리셋은 FrameWise가 독자적으로 만든 파라미터다.
- 업로드한 사진은 서버로 보내지 않고 브라우저 안에서 처리한다.

---

## 슬라이드 14. Film Lab의 기술 구현

### 성능을 위한 선택

- 미리보기는 긴 변을 최대 약 1400px로 제한했다.
- 프리셋 카드는 더 작은 360px 미리보기를 사용한다.
- 다운로드할 때만 더 높은 해상도로 다시 렌더링한다.
- `requestAnimationFrame`으로 슬라이더의 연속 렌더링을 조절한다.
- 사용이 끝난 Object URL과 렌더러 자원을 정리한다.

### 색 처리

- RGB 채널 변환
- 명암 곡선과 블랙 포인트
- 채도, 색온도, tint
- 그림자·하이라이트 색조
- 밝기에 반응하는 결정론적 필름 입자
- 비네팅, 블룸, 할레이션

### 호환성

- 가능한 브라우저에서는 WebGL fragment shader를 사용한다.
- WebGL을 사용할 수 없으면 Canvas2D 방식으로 대체한다.
- CSS `filter`만으로 색감을 흉내 내지 않는다.

- 사진별 대표 구도 가이드 선과 쉬운 설명 추가
- 한국어 번역투를 자연스러운 문장으로 다시 작성
- Image Credits를 화면에 방해되지 않게 추가
- Film Lab의 `원본`과 `결과` 위치 수정
- 사진 경계선과 아래 슬라이더 손잡이의 움직임 동기화

코드가 작동하는 것과 사용자가 자연스럽게 이해하는 것은 다르다.
스크린샷을 보며 “왜 어색한가”를 말로 설명하는 과정이 UI 품질을 크게 높였다.

## Q1. 구도 점수는 AI가 사진을 보고 판단하나요?

아니다. 현재 버전은 사람이 미리 지정한 피사체 좌표, 선, 영역과 사용자의 크롭
좌표를 비교한다. 그래서 계산 과정을 설명할 수 있지만, 새로운 사진을 자동으로
이해하지는 못한다.

## Q2. 그렇다면 프로젝트에서 AI는 어디에 사용했나요?

ChatGPT는 아이디어와 요구사항 정리에, Codex는 조사·코드 구현·테스트·Git 작업에
사용했다. 사용자를 평가하는 핵심 로직은 사람이 설계한 규칙 기반이다.

## Q3. 왜 추천 크롭이 세 개인가요?

사진에는 하나의 절대적인 정답이 없기 때문이다. 같은 목표 구도 안에서도 배경을
넓게 남기거나 피사체를 크게 보여 주는 선택이 모두 가능하다.

## Q4. Film Lab은 실제 필름 색을 정확히 재현하나요?

아니다. 영상과 제품의 일반적 특성에서 상대적인 방향을 참고해 만든 FrameWise
고유의 `Inspired` 스타일이다. 현상, 스캔, 디스플레이에 따라 실제 필름 색은
달라질 수 있다.

## Q5. 업로드한 사진은 서버에 저장되나요?

아니다. 현재 Film Lab은 브라우저 내부의 WebGL 또는 Canvas2D로 처리하며 사진을
외부 서버나 생성형 AI API로 전송하지 않는다.

## Q6. 저작권 문제는 어떻게 관리했나요?

직접 촬영 사진과 재사용 조건이 명확한 Creative Commons 사진을 구분하고,
제작자·원본 주소·라이선스·크롭 변경 사실을 `photos.json`과 Credits 화면에
연결했다. 조사 자료의 사진과 영상 프레임은 프로젝트 자산으로 복사하지 않았다.

## Q7. 가장 많이 개선된 부분은 무엇인가요?

점수가 높고 낮은지만 보여 주던 구조에서, 사진별 목표·세부 점수·측정 근거·가이드
선·복수 추천 크롭을 함께 보여 주는 설명 가능한 학습 구조로 바뀐 점이다.
